# 项目优化方案

基于当前代码分析，从文本切割、召回精准率、上下文管理、模型选择、RAG 流水线等多个维度给出优化建议，按**投入产出比从高到低**排序。

---

## 一、文本切割优化（chunking）

### 现状问题

```yaml
# config/chroma.yml
chunk_size: 200        # 过小，中文 200 字 ≈ 只含 1-2 个句子
chunk_overlap: 20      # 仅 10%，窗口重叠不足
separators: ["\n\n", "\n", ".", "!", "?", "。", "！", "？", " ", ""]
```

200 字的 chunk 对于中文文档来说太小。一个完整的"问+答"条目通常 300-800 字，会被切碎到多个 chunk 中，导致单个 chunk 语义不完整。检索时只能拿到问题片段或答案片段，模型缺乏完整上下文。

### 优化方案

#### 方案 A（推荐，改动最小，效果最明显）

```yaml
chunk_size: 600
chunk_overlap: 120          # 20% 重叠
separators: ["\n\n", "\n", "。", "！", "？", ".", "!", "?", " ", ""]
```

将中文句号 `。` 提前到第二优先级，优先按段落/句子边界切割。

#### 方案 B：语义分块（Semantic Chunking）

替换 `RecursiveCharacterTextSplitter` 为基于 Embedding 相似度的语义分块：

```
1. 先按段落/标题粗分（保留 Markdown 结构）
2. 对每个粗分块计算 embedding
3. 相邻块的 embedding 余弦相似度 < 阈值 → 在此处分隔
4. 确保每个 chunk 语义自包含，不截断句子
```

**建议工具：** `langchain_experimental.text_splitter.SemanticChunker`

#### 方案 C：面向知识库结构化分块

当前知识库是"问答对"格式（`data/扫地机器人100问2.txt`，`data/扫拖一体机器人100问.txt`），应当：

```
1. 按 "Q:" / "A:" / "问：" / "答：" 模式匹配，提取完整问答对
2. 将【完整问答对】作为一个 Document 存入向量库
3. metadata 中标记：type=qa_pair, question=xxx
```

这样检索时命中问题即可带回完整答案，避免 Q 和 A 被切到不同 chunk。

## 二、召回精准率提升（检索质量）

### 现状问题

- 纯稠密检索（Dense Retrieval），单一向量表示
- `k=3` 固定，无法根据 query 动态调整
- 无重排序（Reranking）
- 无查询转换（Query Transformation）
- 无混合检索（Hybrid Search）

### 优化方案

#### 1. 混合检索：稠密 + 稀疏（最高优先级）

```
[用户查询]
    │
    ├──→ DashScopeEmbedding (稠密向量) ──→ ChromaDB 余弦相似度检索
    │
    └──→ BM25 (稀疏关键词) ──→ 倒排索引检索
              │
              └──→ EnsembleRetriever 加权融合
                      │
                      └──→ 返回 Top-N 结果
```

**具体实现：**

```python
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever

# 稠密检索
dense_retriever = vector_store.as_retriever(search_kwargs={"k": 10})

# 稀疏检索（需要先准备分词后的文档）
bm25_retriever = BM25Retriever.from_documents(documents)
bm25_retriever.k = 10

# 加权融合（稠密权重 0.6，稀疏权重 0.4）
ensemble_retriever = EnsembleRetriever(
    retrievers=[dense_retriever, bm25_retriever],
    weights=[0.6, 0.4]
)
```

**为什么有效：** 中文场景下，关键词匹配（如型号"R30 Pro"、故障代码"E2"）对准确率增益很大，纯向量检索容易丢失精确匹配。

#### 2. 重排序 Reranker（高优先级）

初次检索取 Top-10（增加召回深度），用 Cross-Encoder 重排序后取 Top-3：

```
第一次检索 (k=10)  →  粗召回
       │
       ▼
Cross-Encoder Reranker  →  精准排序
       │
       ▼
第二次取 Top-3  →  送 LLM
```

**建议工具和模型：**

| 方案 | 模型 | 说明 |
|------|------|------|
| BGE Reranker | `BAAI/bge-reranker-v2-m3` | 支持多语言，中文效果好，约 1-2GB |
| 轻量 | `maidalen1024/cross-encoder-ms-marco-MiniLM-L-6-v2` | 快速但中文一般 |
| Cohere API | `cohere.rerank-v3` | 云端，无需本地显存，按量付费 |

**预期提升：** NDCG@3 提升 15-25%。Reranker 是"小投入大产出"的经典手段。

#### 3. 查询转换 Query Transformation（中等优先级）

在检索之前，让 LLM 对用户问题进行优化：

```
用户问："机器不充电了"
    │
    ▼
LLM 扩展 → 3 个检索查询:
  1. "扫地机器人无法充电 故障排查"
  2. "扫地机器人充电座 接触不良"
  3. "扫地机器人电池 充不进电"
    │
    ▼
分别检索 → 去重合并结果
```

**实现方式：** 在 `rag_service.py` 中，先调用一次小模型/同模型进行 query 扩展，然后多路检索。

**另一种轻量方式：** Multi-Query Retriever（LangChain 内置）：

```python
from langchain.retrievers import MultiQueryRetriever

retriever = MultiQueryRetriever.from_llm(
    retriever=base_retriever,
    llm=chat_model,
    # 自动生成 3 个变体查询
)
```

#### 4. HyDE（假设性文档嵌入，低优先级）

先让 LLM 根据问题假设性地生成一段理想答案，用这段答案的 embedding 去检索。适用于"用户提问抽象但数据库文档具体"的场景。

#### 5. 元数据过滤（medium priority）

在 `chroma.yml` 中增加知识分类 metadata：

```yaml
# loading 时按文件来源打 metadata tag
metadata:
  category: "故障排除" | "选购指南" | "维护保养" | "常见问答"
```

检索时如果用户问题明显属于某个分类（由 LLM 判断），先按 metadata 过滤再语义检索，减少噪声。

```python
retriever = vector_store.as_retriever(
    search_kwargs={
        "k": 3,
        "filter": {"category": "故障排除"}  # 先按分类过滤
    }
)
```

## 三、上下文管理优化（Context）

### 现状问题

- 检索到的 3 个文档简单拼接，无重要性排序
- 无对话历史管理
- 无上下文压缩（可能塞入大量无关内容）
- 无 "Lost in the Middle" 防护

### 优化方案

#### 1. 上下文压缩 Contextual Compression（高优先级）

```python
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

compressor = LLMChainExtractor.from_llm(chat_model)
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base_retriever
)
```

或者更轻量级：用 LLM 将检索到的文档**压缩提炼**后再送入提示词，大幅减少 token 消耗和噪声。

```
原始检索结果：3 个 chunk × 600 字 = 1800 字
       │
       ▼
LLM 压缩提炼：保留与问题直接相关的 2-3 句
       │
       ▼
压缩后：~300 字 → 送入提示词
```

#### 2. 多轮对话记忆管理

**现状：** `react_agent.py` 每次调用 `execute_stream()` 都是独立会话，没有对话历史。

**优化：**

```python
class ConversationManager:
    def __init__(self, max_history_turns=5):
        self.history = []
        self.max_turns = max_history_turns

    def add_turn(self, query, response):
        self.history.append({"query": query, "response": response})
        # 滑动窗口：保留最近 N 轮
        if len(self.history) > self.max_turns:
            self.history = self.history[-self.max_turns:]

    def build_context(self):
        # 将历史转为摘要或纯文本拼入 system prompt
        context = "\n".join(
            f"用户: {h['query']}\n助手: {h['response']}"
            for h in self.history
        )
        return context
```

**权衡：** 对话轮数多了 token 消耗大。推荐两种策略：
- **短窗口（3 轮）**：直接拼入提示词，简单有效
- **长窗口（10 轮+）**：定期对历史做 LLM 摘要压缩

#### 3. Lost in the Middle 防护

研究表明 LLM 对长上下文中间部分关注度低。优化方案：

```python
def sort_by_relevance(docs, query):
    """将最相关的文档放在开头和结尾"""
    docs_with_score = [(doc, compute_relevance(doc, query)) for doc in docs]
    docs_with_score.sort(key=lambda x: x[1], reverse=True)
    # 重排：最大放开头，次大放结尾，其余中间
    return [docs_with_score[0][0]] + \
           [d[0] for d in docs_with_score[2:]] + \
           [docs_with_score[1][0]]
```

#### 4. 来源引用

在 RAG 输出中添加引用标记，让用户/开发者可追溯：

```
回答：建议每两周清洗一次尘盒滤网。[1]

参考资料：
[1] 《维护保养.txt》第3节 - 滤网保养周期
[2] 《扫地机器人100问.txt》Q47 - 滤网清洗频率
```

实现方式：在 `rag_summarize.txt` 提示词中要求模型在回答时标注 `[n]`，并在 `rag_service.py` 中将 doc metadata 一并返回。

## 四、模型选择优化

### 现状

```yaml
chat_model_name: qwen3-max           # 单一模型做大模型+智能体
embedding_model_name: text-embedding-v4  # DashScope 官方嵌入
```

### 优化方案

#### 1. 嵌入模型替换（Embedding Model）

| 模型 | 维度 | 中文效果 | 建议 |
|------|------|----------|------|
| `text-embedding-v4`（当前） | 1536 | 良好 | 暂时不动 |
| `BAAI/bge-large-zh-v1.5` | 1024 | **优秀** | 替换首选，MTEB 中文榜前列 |
| `intfloat/multilingual-e5-large` | 1024 | 优秀 | 多语言场景 |

**替换成本：** 只需改 `config/rag.yml` + 重新 `load_document()` 一次即可对比。

#### 2. 区分"思考模型"与"生成模型"

**现状：** `qwen3-max` 既做 ReAct 推理（智能体决策）又做 RAG 回答生成。

**优化：**

```
智能体推理（ReAct 循环）→ qwen3-max（需要强推理能力）
       │
       ▼
RAG 总结生成 → qwen3-turbo 或 qwen-plus（更便宜、更快）
       │
       ▼
报告生成 → qwen3-max（需要高质量长文本输出）
```

在 `model/factory.py` 中：

```python
class ModelFactory:
    @staticmethod
    def get_reasoning_model():
        return ChatTongyi(model="qwen3-max", api_key=rag_conf["api_key"])

    @staticmethod
    def get_generation_model():
        return ChatTongyi(model="qwen3-turbo", api_key=rag_conf["api_key"])
```

然后在 `rag_service.py` 中 RAG 总结用 `generation_model`（推理成本降低 50-70%），`react_agent.py` 中 ReAct 循环用 `reasoning_model`。

#### 3. 添加 Reranker 模型

见上一节"召回精准率提升"的重排序部分。Reranker 是独立的 Cross-Encoder 模型，专门用于给检索结果打精排分。

## 五、RAG 流水线整体优化

### 现状架构

```
用户 → Retriever(k=3) → 拼接 Context → Prompt → LLM → 输出
```

### 优化架构（推荐）

```
用户
 │
 ├──→ [查询意图分类] ──→ ① 故障类 → metadata 过滤 "故障排除"
 │                          ② 选购类 → metadata 过滤 "选购指南"
 │                          ③ 通用类 → 全库检索
 │
 ├──→ [查询转换] ──→ 扩展为 3 个相关查询
 │
 ├──→ [混合检索] ──→ 稠密向量 + BM25 加权融合 → 取 Top-10
 │
 ├──→ [Reranker 重排序] → Cross-Encoder 精排 → 取 Top-3
 │
 ├──→ [上下文压缩] → LLM 提炼关键信息 → 降噪
 │
 ├──→ [Chat History] → 拼接对话历史上下文
 │
 ▼
Prompt → LLM → 带引用的回答
```

### 代码架构建议

```python
# rag/optimized_rag.py
class OptimizedRagService:
    def __init__(self):
        self.dense_retriever = ...
        self.bm25_retriever = ...
        self.ensemble = EnsembleRetriever(...)
        self.reranker = BGERReranker(...) if 有 GPU  else APIReranker(...)
        self.compressor = LLMChainExtractor(...)

    def retrieve(self, query):
        # 1. 意图分类 + metadata 过滤
        category = self.classify_intent(query)
        filter_dict = {"category": category} if category else {}

        # 2. 查询扩展
        queries = self.expand_query(query)  # 生成 3 个变体

        # 3. 混合检索
        all_docs = []
        for q in queries:
            docs = self.ensemble.invoke(q, filter=filter_dict)
            all_docs.extend(docs)

        # 4. 去重
        unique_docs = self.deduplicate(all_docs)

        # 5. Reranker 重排
        ranked = self.reranker.rerank(query, unique_docs, top_k=3)

        # 6. 上下文压缩
        compressed = self.compressor.compress_documents(ranked, query)

        return compressed
```

## 六、向量存储优化

### 现状

- ChromaDB 单机文件存储
- 纯余弦相似度检索
- 无分区/分片

### 优化建议

#### 1. ChromaDB 配置调优

```yaml
# 当前
k: 3

# 推荐（搭配 Reranker 使用）
k: 10            # 增大初筛数量
```

#### 2. 分库策略

当前所有知识（故障排除、选购指南、维护保养、问答对）在同一个 collection 中。可考虑：

```
collection "故障排除"  →  专用检索
collection "选购指南"  →  专用检索
collection "常见问答"  →  专用检索
collection "通用"     →  全库检索
```

意图分类后，只检索对应 collection，大幅减少噪声。

#### 3. 生产环境升级路径

```
ChromaDB（当前）→ 单机验证
    │
    ▼
Milvus / Qdrant（下一步）→ 分布式、混合检索原生支持、十亿级规模
    │
    ▼
Elasticsearch（可选）→ 已有 ES 基础设施时，天然支持 BM25 + 向量混合
```

## 七、智能体层优化（Agent）

### 现状问题

- 每次 `execute_stream()` 是独立调用，无状态
- 中间件只有日志和 prompt 切换，没有对话历史管理
- 工具调用最多 5 次硬限制

### 优化方案

#### 1. 工具调用优化

当前 `get_user_location` / `get_user_id` / `get_current_month` 都是随机模拟数据。正式环境中：

- `get_user_location` → 从用户注册信息或 IP 定位获取
- `get_user_id` → 从会话 token 或登录态获取
- `get_current_month` → 改为真实系统时间
- `fetch_external_data` → 对接真实数据库/API

#### 2. 增加工具：用户反馈与学习

```python
@tool(description="记录用户对回答的满意度反馈")
def record_feedback(feedback: str) -> str:
    """记录正面/负面反馈，用于后续优化"""

@tool(description="获取用户历史查询中最相关的FAQ")
def get_user_faq(user_id: str) -> str:
    """基于用户历史，推荐常见问题解答"""
```

#### 3. ReAct 循环优化

将 `middleware.py` 中 5 次工具调用限制改为动态阈值，并加入"工具调用链中断"检测（循环调用同一工具、无效调用链等）。

## 八、评估与可观测性（Observability）

### 现状

完全没有评估机制。无法判断检索质量、回答质量。

### 建议

#### 1. 离线评估集

为当前知识库构建 30-50 组 QA 对（人工标注），每组包含：

```json
{
  "question": "扫地机器人卡在沙发底怎么办？",
  "ground_truth": "检查沙发高度...",
  "expected_chunks": ["故障排除.txt#L42-45", "常见问答.txt#Q15"],
  "category": "故障排除"
}
```

#### 2. 评估指标

| 指标 | 测量对象 | 工具 |
|------|----------|------|
| Hit Rate@K | 检索是否命中正确答案 | 自制 |
| MRR (Mean Reciprocal Rank) | 正确答案的排序位置 | 自制 |
| NDCG@K | 排序质量 | 自制 |
| Faithfulness | 回答是否基于参考 | RAGAS |
| Answer Relevance | 回答是否命中问题 | RAGAS |
| Context Precision | 检索内容精密度 | RAGAS |

#### 3. 检索日志

```python
# 在 rag_service.py 中添加
def log_retrieval(query, retrieved_docs, reranked_docs, latency_ms):
    """记录每次检索的详细信息到日志，用于离线分析"""
    logger.info(json.dumps({
        "query": query,
        "retrieved_count": len(retrieved_docs),
        "reranked_ids": [d.metadata.get("source") for d in reranked_docs],
        "latency_ms": latency_ms,
    }, ensure_ascii=False))
```

## 九、性能优化

### 1. 缓存

```python
from functools import lru_cache
import hashlib

class CachedRagService(RagSummarizeService):
    @lru_cache(maxsize=128)
    def rag_summarize(self, query: str) -> str:
        # 对相同 query 直接返回缓存结果
        return super().rag_summarize(query)
```

### 2. 异步检索

当前是同步串行。可用 `asyncio` 并行执行多个检索查询：

```python
async def parallel_retrieve(queries):
    tasks = [asyncio.to_thread(retriever.invoke, q) for q in queries]
    results = await asyncio.gather(*tasks)
    return list(set(flatten(results)))
```

### 3. 文档加载并发

`vector_store.py` 的 `load_document()` 是逐个文件串行处理。可以改为：

```python
from concurrent.futures import ThreadPoolExecutor

with ThreadPoolExecutor(max_workers=4) as executor:
    futures = [executor.submit(process_file, path) for path in files]
    for future in as_completed(futures):
        ...
```
