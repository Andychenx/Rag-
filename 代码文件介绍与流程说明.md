## 工具层 utils

工具层为项目提供基础设施支撑，包括路径解析、配置加载、文件处理、日志和提示词加载。所有工具模块均不依赖业务逻辑。

### path_tool.py

**文件位置：** `utils/path_tool.py`

**作用：** 为整个工程提供统一的基于项目根目录的绝对路径解析，避免在代码中硬编码相对路径。

**代码介绍：**

```python
def get_project_root() -> str:
    # 通过当前文件的绝对路径向上回溯两级获取项目根目录
    # utils/path_tool.py → utils/ → 项目根目录
    current_file = os.path.abspath(__file__)
    current_dir = os.path.dirname(current_file)
    project_root = os.path.dirname(current_dir)
    return project_root

def get_abs_path(relative_path: str) -> str:
    # 将相对路径拼接到项目根目录上
    return os.path.join(get_project_root(), relative_path)
```

**核心逻辑：**
- `get_project_root()` 利用 `__file__` 的绝对位置，配合两次 `os.path.dirname` 回溯到项目根目录（`utils/path_tool.py` → `utils/` → 根目录）
- `get_abs_path(relative_path)` 拼接根目录和传入的相对路径，返回绝对路径

**流程角色：** 所有需要读取文件的模块（配置加载、文件处理、提示词加载、日志、知识库加载）都依赖此模块获取正确的文件路径。

### factory.py

**文件位置：** `model/factory.py`

**作用：** 使用工厂模式封装 LLM 和嵌入模型的创建，提供统一的模块级模型实例。

**代码介绍：**

```python
class BaseModelFactory(ABC):
    @abstractmethod
    def generator(self): ...

class ChatModelFactory(BaseModelFactory):
    def generator(self):
        # 创建 ChatTongyi 实例
        return ChatTongyi(model=rag_conf["chat_model_name"], api_key=rag_conf["api_key"])

class EmbeddingsFactory(BaseModelFactory):
    def generator(self):
        # 创建 DashScopeEmbeddings 实例
        return DashScopeEmbeddings(model=rag_conf["embedding_model_name"],
                                    dashscope_api_key=rag_conf["api_key"])

# 模块加载时立即创建单例
chat_model = ChatModelFactory().generator()
embed_model = EmbeddingsFactory().generator()
```

**核心逻辑：**
- 使用抽象基类定义工厂接口，子类分别实现 Chat 模型和 Embedding 模型创建
- Chat 模型使用通义千问（`ChatTongyi`），嵌入模型使用 DashScope（`DashScopeEmbeddings`）
- 模型名称和 API 密钥均从 `config/rag.yml` 读取
- 模型实例在模块加载时创建一次，后续以单例形式使用

**流程角色：** 这是所有 AI 能力的核心提供者。`chat_model` 被 `ReactAgent`（智能体主循环）和 `RagSummarizeService`（RAG 总结链）使用；`embed_model` 被 `VectorStoreService`（ChromaDB 检索）使用。

## RAG 层 rag

### vector_store.py

**文件位置：** `rag/vector_store.py`

**作用：** 封装 ChromaDB 向量存储的所有操作（初始化、文档加载、检索器创建），支持 MD5 去重的知识库加载。

**代码介绍：**

```python
class VectorStoreService:
    def __init__(self):
        # 初始化 ChromaDB 客户端
        self.vector_store = Chroma(
            collection_name=chroma_conf["collection_name"],
            embedding_function=embed_model,        # 使用 DashScopeEmbeddings
            persist_directory=chroma_conf["persist_directory"]
        )
        # 初始化文本分割器
        self.spliter = RecursiveCharacterTextSplitter(
            chunk_size=200, chunk_overlap=20,
            separators=["\n\n", "\n", ".", "!", "?", "。", "！", "？", " ", ""]
        )

    def get_retriever(self):
        return self.vector_store.as_retriever(search_kwargs={"k": 3})

    def load_document(self):
        # 1. 扫描 data/ 目录下所有 .txt 和 .pdf 文件
        # 2. 计算每个文件的 MD5
        # 3. 检查 md5.text 文件中是否已有此 MD5（已处理则跳过）
        # 4. 未处理 → 加载文档 → 文本分割 → 存入向量库 → 记录 MD5
        # 5. 异常时记录错误日志并继续处理下一个文件
```

**核心逻辑：**
- `__init__`：初始化 ChromaDB 连接（需在首次使用前调用 `load_document()` 填充数据），设置文本分割器参数（chunk_size=200, chunk_overlap=20，支持中英文分隔符）
- `get_retriever()`：返回封装好的检索器，`k=3` 表示每次检索返回 3 条最相关文档
- `load_document()`：完整的增量知识库加载流程
  - 使用 `listdir_with_allowed_type` 扫描 `data/` 目录下的 `.txt` 和 `.pdf` 文件
  - 使用 MD5 去重避免重复加载已向量化的文档
  - 通过 `pdf_loader` / `txt_loader` 加载原始文档 → `RecursiveCharacterTextSplitter` 分割 → `vector_store.add_documents()` 存入 ChromaDB
  - 处理完的文件 MD5 记录到 `md5.text` 文件

**流程角色：** 这是知识库的入口。首次使用前需运行 `python -m rag.vector_store` 加载知识库文档。后续 `RagSummarizeService` 通过 `VectorStoreService` 的检索器从 ChromaDB 检索相关内容。

### rag_service.py

**文件位置：** `rag/rag_service.py`

**作用：** 实现 RAG 总结链：用户提问 → 检索相关知识 → 构造提示词 → 模型生成回答 → 输出解析。

**代码介绍：**

```python
class RagSummarizeService:
    def __init__(self):
        self.vector_store = VectorStoreService()
        self.retriever = self.vector_store.get_retriever()
        self.prompt_text = load_rag_prompts()       # 加载 RAG 提示词模板
        self.prompt_template = PromptTemplate.from_template(self.prompt_text)
        self.model = chat_model                     # qwen3-max
        self.chain = self.prompt_template | self.model | StrOutputParser()

    def retriever_docs(self, query: str) -> list[Document]:
        return self.retriever.invoke(query)

    def rag_summarize(self, query: str) -> str:
        # 1. 检索相关文档
        context_docs = self.retriever_docs(query)
        # 2. 拼接参考资料文本（含元数据）
        context = "【参考资料1】：xxx | 参考元数据：xxx\n..."
        # 3. 调用 LangChain 表达式语言链
        return self.chain.invoke({"input": query, "context": context})
```

**核心逻辑：**
- 使用 LangChain 表达式语言（LCEL）构建链：`提示词模板 | 模型 | 字符串输出解析器`
- `rag_summarize(query)` 方法：
  1. 从 ChromaDB 检索与 query 最相关的文档（使用 `retriever.invoke`）
  2. 将检索结果格式化为带编号和元数据的参考资料字符串
  3. 将用户问题和参考资料一起传入提示词模板
  4. 模型生成回答后通过 `StrOutputParser` 提取字符串

**提示词模板关键约束（来自 `prompts/rag_summarize.txt`）：**
- 回答必须完全基于参考资料，不编造、不添加未提及内容
- 仅用中文，语气客观简洁
- 严格围绕原始提问总结，不扩展问题范围

**流程角色：** 此服务被 `agent/tools/agent_tools.py` 中的 `rag_summarize` 工具调用。当智能体判断需要补充专业知识时，调用此 RAG 链获取基于知识库的精准回答。

**核心逻辑：**

| 中间件 | 装饰器 | 触发时机 | 副作用 |
|--------|--------|----------|--------|
| `monitor_tool` | `@wrap_tool_call` | 每次工具调用时 | 日志 + 检测 `fill_context_for_report` 设置 context flag |
| `log_before_model` | `@before_model` | 每次模型调用前 | 日志记录状态信息 |
| `report_prompt_switch` | `@dynamic_prompt` | 每次生成提示词时 | 根据 context flag 切换系统提示词 |

**`monitor_tool` 的 context 注入机制是动态提示词切换的关键：**
1. 当模型决定调用 `fill_context_for_report` 并执行时
2. `monitor_tool` 检测到工具名为 `fill_context_for_report`
3. 设置 `request.runtime.context["report"] = True`
4. 下次模型调用前，`report_prompt_switch` 检测到 `report=True`
5. 返回报告生成提示词，模型的行为随之改变

**流程角色：** 中间件在不侵入智能体核心逻辑的情况下，提供了横切关注点的实现。这是整个项目中最巧妙的设计——通过 `fill_context_for_report` 这个空工具作为信号量，驱动提示词切换。

### react_agent.py

**文件位置：** `agent/react_agent.py`

**作用：** 项目主入口，定义 `ReactAgent` 类，组合模型、工具和中间件，提供流式执行接口。

**代码介绍：**

```python
class ReactAgent:
    def __init__(self):
        self.agent = create_agent(
            model=chat_model,                         # qwen3-max
            system_prompt=load_system_prompts(),       # 主系统提示词
            tools=[rag_summarize, get_weather, ...],   # 7 个工具
            middleware=[monitor_tool, log_before_model, report_prompt_switch],  # 3 个中间件
        )

    def execute_stream(self, query: str):
        input_dict = {
            "messages": [{"role": "user", "content": query}]
        }
        # 以流式模式执行，初始 context 中 report=False
        for chunk in self.agent.stream(
            input_dict,
            stream_mode="values",
            context={"report": False}
        ):
            latest_message = chunk["messages"][-1]
            if latest_message.content:
                yield latest_message.content.strip() + "\n"
```

**核心逻辑：**
- `__init__`：通过 `create_agent()` 组装完整智能体，传递模型、系统提示词、工具列表和中间件列表
- `execute_stream(query)`：流式执行入口
  - 将用户查询包装为 `messages` 格式
  - 初始 `context={"report": False}`，智能体处于客服模式
  - 使用 `stream_mode="values"` 逐次获取最新消息
  - 通过 `yield` 逐块返回 AI 的回复内容

**流程角色：** 这是整个应用的执行起点。实例化 `ReactAgent` 并调用 `execute_stream()` 即启动完整的 ReAct 循环。作为演示入口，`__main__` 块中用硬编码的查询"生成我的使用报告"来启动智能体。

## 提示词文件 prompts

### main_prompt.txt

**位置：** `prompts/main_prompt.txt`

**作用：** 智能体的默认系统提示词，定义了 ReAct 智能体的核心行为准则。当智能体处于普通客服模式时使用。

**关键内容：**
- 定义角色为"扫地机器人和扫拖一体机器人的专业智能客服"
- 规定"思考→行动→观察→再思考"的 ReAct 流程
- 详细描述每个工具的使用场景和调用规则
- 约束最多 5 次工具调用，之后仍信息不足则回复"不知道"
- 规定报告生成的固定工具调用顺序（`get_user_id` → `get_current_month` → `fill_context_for_report` → `fetch_external_data`）
- 要求每次调用工具前输出自然语言思考过程

### report_prompt.txt

**位置：** `prompts/report_prompt.txt`

**作用：** 报告生成模式下的系统提示词。当 `fill_context_for_report` 被调用触发动态提示词切换后，模型使用此提示词。

**关键内容：**
- 定义角色为"专业的扫地/扫拖机器人报告写手"
- 列出报告模式可用的工具（与客服模式相同的工具集）
- 要求根据查询回来的信息生成 MarkDown 格式的使用报告
- 标题固定为"扫地机器人使用情况报告与保养建议"
- 要求根据使用情况给出具体建议，禁止直接输出原始数据

### rag_summarize.txt

**位置：** `prompts/rag_summarize.txt`

**作用：** RAG 总结链使用的提示词模板，包含 `{input}` 和 `{context}` 两个占位符。

**关键内容：**
- 定义角色为"基于参考资料总结的 AI 助手"
- 约束回答必须完全基于参考资料，不编造不添加
- 仅用中文回答，语气客观简洁
- 严格围绕用户原始提问总结
- 仅输出纯文本内容，不封装为任何结构

## 整体执行流程

### 流程总览

```
用户输入
    │
    ▼
ReactAgent.execute_stream(query)  ───  context={"report": False}
    │
    ▼
create_agent() 组装智能体:
    ├── 模型: qwen3-max (ChatTongyi)
    ├── 系统提示词: main_prompt.txt（默认，客服模式）
    ├── 工具: 7 个 @tool 函数
    └── 中间件: monitor_tool, log_before_model, report_prompt_switch
    │
    ▼
agent.stream() 开始 ReAct 循环
    │
    ├── 1. 模型推理（使用当前系统提示词）
    │       ↓
    ├── 2. 模型返回：输出回答 或 调用工具
    │       ↓
    ├── 3. (若调用工具) 被 monitor_tool 拦截 → 记录日志
    │       │                            ↓
    │       │                   如果是 fill_context_for_report：
    │       │                   context["report"] = True
    │       ↓
    ├── 4. (若调用工具) 执行工具函数 → 结果返回模型
    │       ↓
    ├── 5. 再次模型推理前，report_prompt_switch 检查 context
    │       - report=False → main_prompt.txt（客服模式）
    │       - report=True  → report_prompt.txt（报告生成模式）
    │       ↓
    └── 6. 重复 1-5 直到模型给出最终回答
            │
            ▼
       流式输出最终回答
```

### 模式一：普通客服咨询（默认）

1. 用户提问如"扫地机器人在我所在的地区的气温下如何保养？"
2. 智能体使用 `main_prompt.txt`（客服模式）
3. 模型依次调用：`get_user_location()` → `get_weather(city)` → `rag_summarize(query)`
4. 结合天气信息和 RAG 检索结果给出保养建议

### 模式二：报告生成（动态切换）

1. 用户提问如"生成我的使用报告"
2. 智能体使用 `main_prompt.txt`（客服模式），按约束执行固定流程：
   - `get_user_id()` → 获取用户 ID
   - `get_current_month()` → 获取当前月份
   - `fill_context_for_report()` → **调用时触发中间件将 `context["report"]=True`**
3. 下一次模型推理前，`report_prompt_switch` 检测到 `report=True`
4. **系统提示词动态切换为 `report_prompt.txt`**（报告生成模式）
5. 模型在报告模式下继续：
   - `fetch_external_data(user_id, month)` → 获取使用记录
6. 模型根据使用记录生成 MarkDown 格式的使用报告

### RAG 知识库加载流程（一次性前置操作）

```
python -m rag.vector_store
    │
    ▼
扫描 data/ 目录下所有 .txt 和 .pdf 文件
    │
    ▼
对每个文件：
    ├── 计算 MD5
    ├── 检查 md5.text → 已处理则跳过
    ├── 未处理 → 加载文件 → 文本分割(RecursiveCharacterTextSplitter)
    ├── 存入 ChromaDB (embedding 由 DashScopeEmbeddings 生成)
    └── 记录 MD5 到 md5.text
    │
    ▼
完成知识库加载，可开始使用 RAG 工具
```
